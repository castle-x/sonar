## 💡 关键决策

### 为什么现在是 72% 而非更高？

1. **评分系统完全缺失**（应有权重 15%）
   - 理由：复杂度最高，需要完整的 service + repo 层
   - 状态：设计完整，实现未开始

2. **快照高级功能部分缺失**（权重 8%）
   - 指标查询 API（需要解压和反序列化）
   - 快照列表筛选（需要后端 MongoDB 查询优化）

3. **前端数据绑定不完整**（权重 5%）
   - SnapshotDetailPage 的评分显示依赖后端
   - TapProxyPage 配置代理未实现

### 为什么评分系统是 P0？

1. **无法显示快照评分**
   - 用户无法了解压测质量
   - 用户无法对比不同快照

2. **无法实现快照重评**
   - 评分模板管理功能完全缺失

3. **前端页面无法完成**
   - SnapshotDetailPage 右侧评分分解栏无数据

### 为什么不实现 TapProxyPage？

1. **次要功能**（P1，可先完成核心）
   - Tap 本地 UI (:9090) 已完整
   - view 中的 tap 代理是 "方便管理" 而非 "必需"

2. **技术复杂度高**
   - 需要复用 sonar-tap 的 React 组件
   - 涉及跨项目组件通信

3. **后端已完整**
   - httputil.ReverseProxy 已实现
   - 仅缺前端 UI 包装
